**Reviewed file:** `app/database/graph_schema.py` (1,643 lines)

**Author:** Daniel Fontaine, SFBRN Accessibility Lead
**Review date:** May 13, 2026
**Reviewer note:** This document captures an assessment of the asset ownership modeling that was added to the schema. It is written to be used in a separate working session, so it restates the relevant context rather than assuming it.

---

## Context

The schema models the CSU Accessible Technology Initiative (ATI) compliance domain. Earlier versions modeled Year Success Evidence (YSE), Implementation Types (Process, Project, Procedure, Service, InternalPolicy, Guidance, Tracking), Documentation, People, Organizational Units, Academic Years, Status Levels, and Success Indicators. The recent push added an **asset ownership layer** to represent the ICT assets the institution is accountable for, who stewards them, and how their accessibility is remediated.

This work operationalizes a policy framework with these core ideas (preserved here so the review stands alone):

- **The legal duty runs to the institution as a whole**, not to specific offices. How responsibility is distributed internally is a question of institutional design.
- **Two parallel, non-substituting duties:** reasonable accommodation (individualized, person-centered, via disability services and HR) and program accessibility (systemic, asset-centered, via the units that build and operate ICT).
- **Three rules for distributing responsibility:**
  1. Responsibility sits with the party closest to the capacity to remediate (configuration/authorship/maintenance authority for an asset; institutional relationship to the person for an accommodation).
  2. Where the closest party cannot remediate, responsibility rises to the institution.
  3. Where program accessibility requires coordination across multiple owners, coordination sits with SFBRN ATI.
- **Line authority vs. functional authority:** asset owners hold line authority (can configure/author/maintain); the accessibility coordinating function (SFBRN ATI) holds functional authority (accountable for an institutional outcome it does not directly control). The separation is a feature — a coordinating function that owned the assets it evaluated couldn't credibly evaluate them.
- **Emergence / collective action:** institutional program accessibility is an emergent property no single owner produces alone; coordination must be produced, not assumed.

## What the New Schema Does

### New nodes and constructs

- `Asset` — a logged unit of ICT, defined from the remediation side (the unit at which remediation authority is coherent; where authority splits, the asset splits).
- `TAAP` — Temporary Alternate Access Plan; the institution's response when full conformance is not achievable. Asset-scoped, time-bound, annually reviewed, anchored in Title II §35.205. Replaces the older EEAAP.
- `OrgUnit` — new shared supertype for `Department` and `College` (both now inherit from it), so asset stewardship edges can target a unit, not only a Person.
- Also present from related work: `UniversalDescriptor`, `CampusPlan`, `WorkingGroupPlan`, `ProgressUpdate`, `YseProgressRel`, and a `trajectory` property on progress updates. These are adjacent to the asset work, not part of it.

### The central design decision: stewardship vs. remediation accountability

This is the strongest part of the design and the thing to preserve.

- **§508 stewardship** lives on the `Asset` node: `procured_by`, `developed_by`, `maintained_by`, `used_by`, each targetable to a `Person` OR an `OrgUnit`. This answers "who stands in a §508 relationship to this thing."
- **Remediation accountability** lives on the Implementation nodes (Process, Project, Procedure, Service) via the `remediates` edge to an Asset plus `owned_by` to a Person. This answers "who is accountable for the work that keeps it accessible."

These are distinct because "ownership" in the policy framework was carrying two jobs that are not the same: being the §508 duty-holder, and being the party accountable for remediation. Example: the Library stewards a licensed database (procures and uses it), but the remediating work might be a Process owned by SFBRN ATI plus vendor engagement. The schema represents that without conflating the two.

### Rule 2 expressed as graph structure

From the Asset docstring:

> An asset that is stewarded yet has no remediating implementation is the modeled signal that responsibility has elevated to the institution (Title II §35.205 / the responsibility heuristic).

The elevation is not a stored property — it is the **absence** of a `remediates` edge into a stewarded asset. This makes "responsibility has risen to the institution" a queryable graph pattern: find every stewarded asset with no incoming `remediates` edge. That query is, in effect, an institutional risk report surfacing exactly the §35.205 elevation cases.

## What Was Done Well

1. **Stewardship/remediation split** — sharper than treating "ownership" as one thing. Lets the graph represent the common real case where the steward and the remediating party differ.

2. **Rule 2 as edge absence** — the §35.205 elevation is modeled as a missing edge, which is both philosophically consistent with the framework (model the absence) and operationally useful (one Cypher pattern surfaces all elevation cases).

3. **Shared rel-type for Person-OR-OrgUnit stewardship** — `procured_by` and `procured_by_unit` are separate neomodel definitions sharing one Neo4j rel-type string, so raw Cypher like `(:Asset)-[:maintained_by]->(holder)` catches both holder types. Correct handling of a polymorphic target neomodel doesn't natively support.

4. **Composite identity with scope baked in** — `asset_identifier` ("canvas-sfsu" vs "canvas-systemwide") makes scope part of identity, so one nominal system resolves into distinct assets where remediation authority sits at different scopes. Answers the campus/regional/systemwide scoping question and mirrors existing `plan_identifier` / `year_identifier` conventions.

5. **`uses_tool` vs. `remediates` as distinct edges** — a Process can remediate Asset A while using Asset B (Pope Tech, Equidox) as a tool. Separate rel-types keep "what keeps Canvas accessible" and "what depends on Pope Tech" as different queries, and a tool's own accessibility status doesn't get confused with the assets it fixes.

6. **Asset documentation kept off the evidence backbone** — `describes_asset` (VPAT/ACR/spec) is a distinct rel-type from `is_documented_by`, so asset documentation isn't swept into year-scoped evidence queries. A VPAT describes an asset; it isn't evidence for a success indicator in a given year.

7. **TAAP on the evidence backbone, Asset deliberately off it** — TAAP has `is_evidence_for → YearSuccessEvidence` (a real artifact proving interim coverage); the Asset does not. The asset's status reaches YSE only through remediating implementations or a covering TAAP. The explicit comment at the YSE node ("Assets are NOT evidence... there is no direct Asset → YSE edge") documents this at the point of temptation.

8. **`remediates`/`uses_tool` correctly limited to the four "doing" implementations** — present on Process, Project, Procedure, Service; absent from Guidance (references other implementations), Tracking (observes), and InternalPolicy (mandates). Only work that directly acts on an asset carries `remediates`.

## Open Items Worth a Second Look

Ordered roughly by priority.

### 1. Where does monitoring/scanning live? (settle before populating asset data)

`remediates`/`uses_tool` are absent from `Tracking`. But automated scanning (Pope Tech sweeps, D-PAR) arguably *uses* an asset as a tool without remediating anything. If scanning is modeled as `Tracking`, it currently cannot express `uses_tool`. Two clean options:

- Model monitoring as a `Process` (which has `uses_tool`), or
- Give `Tracking` a `uses_tool` edge but not `remediates`.

This matters now because Pope Tech and D-PAR are live infrastructure and the decision determines how they wire into the graph. **This is the one to resolve first.**

### 2. No "primary" or typed discriminator on stewardship edges

The four capacities are four named edges, which is clean, but within (say) `maintained_by`, if three units each maintain a piece of one asset, the graph can't yet say which is *primary* or which piece each holds. The framework's "closest to the capacity to remediate" sometimes needs a primary holder. If "who is the responsible steward" queries start returning ties, add an `{is_primary: true}` property via a small `StructuredRel` on the stewardship edges. Not needed yet; flagged so it can be absorbed later without rework.

### 3. Functional authority (SFBRN ATI) is modeled implicitly, not positively

The framework's sharpest claim is that ATI holds functional authority *without* line authority. The schema honors this implicitly: ATI appears as the `owned_by` Person/holder on remediating Processes and Services (its actual work) and as steward of the *tools* it operates (Pope Tech), but does *not* appear in the stewardship edges of the assets it coordinates. So functional authority is currently "shows up as owner of coordinating work, absent from stewardship of coordinated assets."

This is elegant and consistent with the emergence framing, but it is *inferred*, not asserted. If an auditor asks "where does the graph say ATI is accountable for the whole?", the current answer is "nowhere directly; it emerges from the edges." Decide whether to add a positive assertion (e.g., a relationship stating ATI is accountable for aggregate program accessibility at a Campus/scope) or to keep it purely emergent. This is a genuine design choice, not a bug.

### 4. TAAP `signed_by` is flat (no role)

The policy framework established a three-tier signature structure (Department Chair/Manager + Dean/Division VP + ADA Compliance Officer), and TAAP `outcome` captures the risk tier. But `signed_by → Person` does not capture *which role* signed. To verify "did this TAAP get all required signatures for its risk tier," put a role property on the signature edge (small `StructuredRel`). The kind of thing an OCR review would actually ask about. Later concern.

### 5. No conformance status on the Asset itself

The Asset records what it is and who stewards it, but not its accessibility status (conforms / partially / does not / under TAAP). Likely deliberate — status is point-in-time and can be inferred from "has a remediating implementation / has an active TAAP / has neither." Trade-off: a stored status can drift out of sync with edge reality; an inferred status is always current but costlier to query. If a direct "show me all non-conforming assets" query is wanted without inferring from edge patterns, add a status property or a `StatusLevel`-style relationship on Asset. Make this a deliberate decision rather than leaving it implicit.

## Reference: Key Edge Inventory (asset layer)

**On `Asset`:**
- Stewardship (to Person or OrgUnit, shared rel-types): `procured_by`, `developed_by`, `maintained_by`, `used_by`
- Provenance: `supplied_by` → Vendor
- Scope anchor: `at_campus` → Campus (empty for systemwide; `scope` property carries it)
- Remediation (reverse): `remediated_by_{processes,projects,procedures,services}` ← Implementation `remediates`
- Tool use (reverse): `used_as_tool_by_{processes,projects,procedures,services}` ← Implementation `uses_tool`
- TAAP coverage (reverse): `covered_by_taap` ← TAAP `covers_asset`
- Documentation (distinct from evidence): `described_by` → Document, `described_on` → Webpage, `notes` → Note
- Identity: `asset_identifier` (unique), `title` (indexed, not unique), `asset_class`, `scope`

**On Implementation nodes (Process, Project, Procedure, Service):**
- `remediates` → Asset
- `uses_tool` → Asset
- `owned_by` → Person
- `is_evidence_for` → YearSuccessEvidence

**On `TAAP`:**
- `covers_asset` → Asset
- `owned_by` → Person
- `signed_by` → Person
- `is_evidence_for` → YearSuccessEvidence
- standard `supporting_{documents,webpages,notes,messages}` via DocumentedByRel
- properties: `outcome` (equally_effective | non_equal_alternative | referral), `effective_date`, `review_due`, `active`

**Choice lists (from `app.data_config`):** `asset_classes` (institutional_system | employee_content | third_party_service | infrastructure), `asset_scopes` (systemwide | regional | campus | vendor), `taap_outcomes`, `trajectory_choices`.

## Suggested Next Steps

1. **Resolve the monitoring/scanning question (#1)** before populating asset data, since it affects how Pope Tech and D-PAR wire in.
2. Decide whether functional authority should be asserted positively or left emergent (#3) — this is the most conceptually significant open question and ties directly to the policy framework's central claim.
3. Defer #2 (primary steward), #4 (signature roles), and #5 (asset conformance status) until a query need forces them; all can be added incrementally.
4. Plan the back-label migration noted in the `OrgUnit` docstring (adding the `:OrgUnit` label to existing `:Department` / `:College` nodes so OrgUnit-label traversal finds them).
